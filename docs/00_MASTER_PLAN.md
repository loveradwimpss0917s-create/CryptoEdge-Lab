# CryptoEdge Lab — 完全設計書 (Master Plan)

> **Document version**: 1.0.0 / 2026-07-02
> **対象読者**: 実装担当AI (Sonnet 5) および開発者
> **性質**: 本書は設計書であり、コードは含まない。実装は本書群 (docs/00〜12) のみを入力として行えること。

## 1. プロダクト定義

### 1.1 一文定義

**CryptoEdge Lab は、市場データから未知の統計的優位性 (Edge) を発見し、標準化されたプロトコルで検証し、劣化を監視しながらライフサイクル管理する、個人向けクオンツ研究プラットフォームである。**

Trading Tool ではない。注文執行機能は持たない (V3 でもペーパートレードまで)。
成果物は「検証済み Edge のポートフォリオ」と「毎日の研究ブリーフィング」である。

### 1.2 プラットフォームが毎日行うこと (Daily Loop)

```
 収集 → 品質検査 → 特徴量計算 → レジーム更新 → 全Edge再評価 → 劣化検知
   → 新候補スクリーニング → AIブリーフィング生成 → 研究者が朝5分で意思決定
```

### 1.3 非目標 (Non-Goals)

| 非目標 | 理由 |
|---|---|
| 自動売買・注文執行 | 研究プラットフォームであり、執行は責任範囲外。API鍵の秘匿・執行品質は別問題系 |
| 既存テクニカル指標のカタログ化 (RSI/MACD/ICT/Wyckoff 等) | 公知の手法の陳列は Edge ではない。特徴量の素材として一部を使うに留める |
| マルチユーザ SaaS 化 (V1-V2) | 単一研究者向けに最適化。認証は Cloudflare Access で足りる |
| ティックデータのリアルタイム処理 (V1) | Workers の制約と ROI から、V1 は 1分足以上の解像度に限定 |

## 2. 参考資料 (PDF: 54 Edge Candidates) の批判的評価

添付 PDF「CryptoEdge Research Themes: 54 BTCUSDT Alpha Candidates and Structural Edge Analysis」を精読した。本プラットフォームは PDF の再現ではなく、その弱点を構造的に解決するシステムとして設計する。

### 2.1 優れている点 (継承する)

1. **文献に接地している**: Saggu (2022)、Chen, Deng & Nie (2024)、Urquhart (2017)、Pyo & Lee (2020)、Takaishi (2021)、QuantPedia 等、一次文献への参照があり、単なる思いつきの羅列ではない。→ 本システムでも Edge には `evidence` (文献・出典) フィールドを必須化する。
2. **「なぜ Edge が存在し続けるのか」の構造分析**: 清算カスケード=強制的流動性需要、CME ギャップ=市場時間ミスマッチ、VRP=ヘッジ需要、といった経済的根拠 (economic rationale) を各候補に付けている。→ `hypothesis` / `rationale` を Edge スキーマの必須カラムにする。
3. **コスト後 Sharpe > 1 などの明示的な採用閾値**: 「採用/保留/棄却」の判断基準が数値で書かれている。→ 評価プロトコル (docs/05) の Verdict 閾値として体系化する。
4. **反証の併記**: Griffin & Shams (2020) に対する Wei (2018) の反証、Edge 015 の Mar 2020 外れ値依存など、都合の悪い証拠を隠していない。→ Edge Dossier に「反証・注意点 (counter-evidence)」セクションを必須化。
5. **P0〜P3 の優先順位付けと段階的着手戦略**: ROI 意識がある。→ ロードマップの初期シード Edge として P0 の 5 件 (021, 018, 006, 031, 013) をそのまま採用する。

### 2.2 不足している点 (本設計で構造的に解決する)

| # | 問題 | 帰結 | 本設計での解決 |
|---|---|---|---|
| W1 | **多重検定の未補正**。54 候補を個別に p<0.05 で検定すると期待値で約 2.7 個の偽陽性が「有意」になる。星評価は主観であり統計的制御がない | 偽 Edge の量産 | 評価プロトコルに **FDR 制御 (Benjamini–Hochberg)**、**Deflated Sharpe Ratio (Bailey & López de Prado 2014)**、**試行回数レジストリ** (何回バックテストを回したかを DB に記録し、補正に使う) を組み込む (docs/05 §4) |
| W2 | **静的なリスト**。54 件は 2026 年時点のスナップショットであり、新候補を生成する仕組みがない | 発見が止まる | **Edge Discovery Engine** (docs/04)。特徴量 × 変換 × 条件付けの文法による候補の系統的生成 + AI による文献駆動の仮説生成 |
| W3 | **評価手順が候補ごとにバラバラ**。「勝率>55%」「Sharpe>0.8」「MSE改善」など基準が混在し、横比較できない | 恣意的な採用判断 | **標準化 Edge Evaluation Protocol (EEP)**。全 Edge が同一のパイプライン (コストモデル → Walk-Forward → Permutation → Bootstrap → レジーム別) を通る (docs/05) |
| W4 | **劣化 (decay) の監視がない**。McLean & Pontiff (2016) が示す通り、公表されたアノマリーはリターンが平均 58% 減衰する。PDF の候補の多くは公開文献由来であり、既に劣化している可能性が高い | 死んだ Edge を運用し続ける | Edge ライフサイクル状態機械に **DECAYING / RETIRED** 状態と、CUSUM による自動劣化検知を組み込む (docs/05 §7) |
| W5 | **ポートフォリオ視点の欠如**。Edge 006 (清算リバウンド) と 007 (Funding 逆張り) は同一のレバレッジ解消現象の別측面であり、強く相関するはず。個別評価だけでは重複リスクを見逃す | 分散していない「見せかけの複数 Edge」 | **Edge 相関マトリクス** (シグナル重複率・リターン相関) を評価に含め、ポートフォリオ限界貢献 (marginal Sharpe) で採否を判断 (docs/05 §8) |
| W6 | **レジーム条件付けが場当たり的**。「ベア相場では SMA フィルタ」など個別対応に留まる | 汎化しない | プラットフォーム共通の **レジームラベル** (トレンド/ボラ/流動性の 3 軸、HMM + ルールベース) を全評価に自動適用 (docs/04 §6) |
| W7 | **データ品質・ポイントインタイム性への言及がない**。オンチェーン指標や ETF フローは後日修正される。修正後データでバックテストすると look-ahead bias | 再現不能な好成績 | **as-of (ingested_at) 二時制データモデル**と、データ品質検査を収集パイプラインに組み込む (docs/02 §2, docs/03 §6) |
| W8 | **実行環境の検討がない**。GARCH・XGBoost・Bootstrap 1万回は Cloudflare Workers 上では現実的に動かない | 実装不能な設計 | **二層コンピュート**: Workers = 収集/配信/軽量統計、GitHub Actions (Python) = 重量統計。結果は R2/D1 に還流 (docs/01 §3) |
| W9 | **星評価 (独創性★3 等) が主観で更新されない** | 評価が腐る | 主観評価は `priors` として保持しつつ、実測メトリクスが蓄積され次第、実測値が優先される設計 (docs/02 edges テーブル) |
| W10 | **古い考え方**: 「手法を貯めるほど強くなる」という暗黙の前提。現代クオンツの合意は「候補は無限に湧く。ボトルネックは検証の厳密さと劣化管理」 | 誤った投資配分 | プラットフォームの中心を「候補リスト」ではなく「**検証パイプラインとライフサイクル管理**」に置く。UI も Edge Board (状態機械) を中核にする (docs/06) |

## 3. 設計原則 (全ドキュメント共通の憲法)

実装時に判断に迷ったら以下に従う。番号が小さいほど優先。

1. **再現性が最優先 (Reproducibility First)**: すべての評価結果は「Edge定義バージョン + データセットハッシュ + パラメータ + 乱数シード」から決定論的に再現できること。再現できない結果は保存しない。
2. **ポイントインタイム原則**: バックテストで参照してよいのは「その時点で入手可能だったデータ」のみ。全時系列テーブルは `ts` (事象時刻) と `ingested_at` (取得時刻) の二時制を持つ。
3. **統計的誠実さ**: 多重検定補正なしの p 値を UI に表示しない。試行回数は自動記録され、ユーザーが隠せない。
4. **コストは常に込み**: コスト前のバックテスト結果は「参考値」ラベル付きでしか表示しない。既定コストモデル: taker 手数料 0.04% + スリッページ 0.02%/side (設定可能)。
5. **Workers に重い計算をさせない**: CPU 時間 30 秒を超えうる処理 (GARCH, ブートストラップ, 木モデル学習) は GitHub Actions の研究ワーカーで実行する。
6. **無料 API から始め、抽象化して有料へ**: すべてのデータソースはアダプタインターフェイス越しに扱い、CryptoQuant/Glassnode/Amberdata 等の有料化はアダプタ追加で済むこと (docs/03 §7)。
7. **削除しない、状態遷移する**: Edge・評価結果・データは物理削除せず、状態 (RETIRED, superseded) で管理する。研究の失敗履歴こそ資産。
8. **UI は「次に何をすべきか」を提示する**: 数値の陳列ではなく、毎画面に必ず「推奨アクション」領域を持つ (docs/06 §1)。
9. **AI は仮説生成と要約に使い、判定には使わない**: 採用/棄却の Verdict は決定論的な統計パイプラインのみが下す。AI の出力は常に「候補」「ドラフト」として人間の確認を経る (docs/07 §1)。

## 4. システム全体像 (要約)

詳細は docs/01。ここでは全体の骨格のみ示す。

```
┌─────────────────────────── Cloudflare ────────────────────────────┐
│                                                                     │
│  [Workers: api]  ← Hono REST API + 静的アセット配信 (React SPA)     │
│       │ 読取                                                        │
│  [D1: cryptoedge]  メタデータ・集計済み時系列・Edgeレジストリ        │
│  [KV]  APIキャッシュ・最新スナップショット・ジョブロック             │
│  [R2: lake]  生データ Parquet/NDJSON・レポート・モデル成果物         │
│       ↑ 書込                                                        │
│  [Workers: ingest]  Cron (1m/5m/1h/1d) → Queues → fetcher 群        │
│  [Queues: ingest-tasks / dlq]                                       │
│  [Workers AI + AI Gateway]  要約・仮説生成・レポート                 │
└─────────────────────────────────────────────────────────────────────┘
              │ dispatch (repository_dispatch)         ↑ 結果還流 (認証付き API)
┌─────────────────────────── GitHub ────────────────────────────────┐
│  [GitHub Actions: research-worker]                                  │
│   Python (pandas/statsmodels/scikit-learn/xgboost/arch)             │
│   - Edge Evaluation Protocol 実行 (WF/CPCV/Permutation/Bootstrap)   │
│   - レジーム推定 (HMM)・Change Point・Discovery スクリーニング       │
│   - R2 から Parquet 読取 → 結果 JSON を Worker API へ POST          │
└─────────────────────────────────────────────────────────────────────┘
```

**最重要のアーキテクチャ判断**: 統計計算を Cloudflare 内で完結させない。
理由: (a) Workers は Python 科学計算スタックが使えず、JS 移植は品質リスクが大きい。(b) Cron Worker の CPU 上限 (有料プランで 5 分/呼出し, 30 秒既定) に対し、54 Edge × Walk-Forward × 1000 回 Permutation は桁違いに重い。(c) GitHub Actions は既に接続済みで、private repo でも月 2,000 分の無料枠があり、夜間バッチ (1 日 30–60 分) に十分。
この判断により「Cloudflare = データプレーン + 配信」「GitHub Actions = 研究コンピュートプレーン」という明確な責務分離が得られる。

## 6. 用語集 (全巻共通)

| 用語 | 定義 |
|---|---|
| **Edge** | 検証可能な形式で記述された市場の統計的優位性の仮説。`シグナル定義 + 対象市場 + 保有期間 + 経済的根拠` の組 |
| **Edge Version** | Edge のパラメータ・シグナル定義の不変スナップショット。評価は常に特定バージョンに対して行う |
| **Run** | ある Edge Version × データセット × プロトコル設定での 1 回の評価実行 |
| **EEP** | Edge Evaluation Protocol。全 Edge 共通の標準評価パイプライン (docs/05) |
| **Verdict** | EEP が下す判定: `ADOPT` / `WATCH` / `REJECT` |
| **Dossier** | 1 つの Edge に紐づく全証拠 (仮説・文献・Run 履歴・反証・相関) の閲覧単位 |
| **Regime** | 市場状態ラベル。trend (up/down/range) × vol (low/high/extreme) × liquidity (normal/stressed) |
| **Feature** | 特徴量。生データから決定論的に計算される時系列 (例: funding z-score 30d) |
| **Dataset Snapshot** | 評価に使うデータの不変スナップショット (R2 上の Parquet 群 + manifest のハッシュ) |
| **Trial Registry** | 多重検定補正のために全バックテスト試行を記録する台帳 (`eval_runs` が兼ねる) |
| **Paper Signal** | 採用済み Edge が実データ上でリアルタイムに出す仮想シグナル。約定はシミュレーション |
