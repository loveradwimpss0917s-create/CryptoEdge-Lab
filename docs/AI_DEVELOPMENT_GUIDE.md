# AI Development Guide — AI共同開発の運用規約 (2026-07-12)

> **位置づけ**: 本書は「どのAIが・何を・どう進めるか」の正典。設計の中身は番号付き docs、
> セッション状態は docs/HANDOFF.md、本書は**規約**のみを持つ。
> 本書と番号付き docs が矛盾したら本書 §6 (設計変更ルール) に従い ADR で解消する。

## 1. 役割分担

| 役割 | 担当AI | 責務 | やらないこと |
|---|---|---|---|
| **CTO / Architect** | Fable (メイン) | 設計レビュー・監査・ADR 起票・タスクカード作成 (docs/19)・Phase 優先順位の決定・docs/21 級の構想文書・Sonnet 成果物の受入判定 | 原則コードを書かない。書く場合は「設計の実証に必要な最小限」と明記する |
| **Senior Software Engineer** | Sonnet (サブ) | docs/19 カードの実装・テスト・デプロイ検証・実行ログ追記・HANDOFF 更新 | 設計変更 (カードの「設計判断 (確定)」を覆すこと)。必要になったら実装を止めて HANDOFF の Open Questions に起票する |
| 人間 (オーナー) | — | Phase 承認・本番UI での受入確認・secrets/外部データ提供・main 反映の承認 | — |

**役割は「モデル名」ではなく「振る舞い」で定義する**。別のAI (別ベンダー含む) が
セッションを引き継ぐ場合も、この2役のどちらかを宣言して同じ規約で動けばよい。

## 2. 文書体系 — どこに何を書くか

番号付き docs (00〜21) は**設計の正典**で、相互参照 (「docs/05 §2」形式) がコード・コミット・
実行ログに数百箇所ある。**ファイル名変更・ディレクトリ再編は禁止** (参照網が壊れる)。
一般的な docs 構成案との対応は以下のとおり:

| 一般的な構成案 | 本リポジトリでの実体 |
|---|---|
| vision.md | docs/00 (思想) + docs/21 (Research OS 構想) |
| roadmap.md | docs/18 (正典) + docs/09 (バージョン戦略の上位原理) |
| architecture.md | docs/01 (+ docs/13 無料枠正典) |
| glossary.md | docs/00 §7 用語集 |
| adr/ | **docs/17 §6 の ADR 台帳** (単一表で管理) |
| specs/ | docs/02〜08, 14, 16, 20, 21 (領域別設計書) |
| tasks/ | docs/19 (カード + 実行ログ) |
| phases/ | docs/18 §3 (Phase 表) + docs/19 の Phase 節 |
| handoff/ | **docs/HANDOFF.md** (常に1枚、履歴は git log) |
| ai/ | 本書 + docs/07 (プロダクト側のAI統合設計) |

**書き込み先の規則**:
- 新しい設計判断 → 該当する番号付き docs に日付付き追記 + 重要なら docs/17 §6 に ADR 1行
- 新しいタスク → docs/19 にカード + docs/18 §3 の表に1行
- 実装の結果 → docs/19 該当カードに「実行ログ (日付, 役割)」
- セッションの現在地 → docs/HANDOFF.md を**上書き** (追記ではない。履歴は git が持つ)
- 監査・負債の発見 → docs/17 に日付付き追記

## 3. Phase 運用とカードのライフサイクル

1. **着手**: HANDOFF の Current Card を確認 → docs/19 の該当カードと「関連docs」を読む
2. **実装** (Engineer): カードの「設計判断 (確定)」に従う。判断が書かれていない選択は
   「既存パターンの踏襲」(docs/19 §0-2) で埋め、それも無理なら Open Questions 化して停止
3. **検証**: docs/19 §0-4 の完了手順 (typecheck/test/lint 全緑 → push → deploy 確認 → 本番読み取り検証)
4. **記録**: docs/19 カードに実行ログ → docs/18 §3 の表の状態更新 → HANDOFF 更新
5. **受入** (CTO): 次の Fable セッションが受入条件を実測で確認。不合格なら差し戻しカード化
6. Phase 内の全カード完了 → CTO が Phase レビュー (docs/17 に追記) → 次 Phase へ

**1カード = 1セッション**を原則とする (docs/19 冒頭)。カードが1セッションで終わらない
規模だと判明したら、それ自体を設計不備として分割カードを CTO が切る。

## 4. 実装ルール

docs/19 §0 (共通実装規約) が正典。要点の再掲 + 本書での追加:

- 新パターンを発明しない (アダプタ/ルート/ジョブの追加は既存方式に従う — docs/19 §0-2)
- edge_version / eval の作成は必ず API 経由。**D1 直接 INSERT 禁止** (docs/19 §0-3)
- スキーマ変更は migration ファイルのみ (連番、既存 migration の書き換え禁止)
- TS↔Python の二重実装 (DSL・metric名) は golden fixture (packages/schema/fixtures/) を
  必ず両側テストで共有する (docs/11 §4)
- コメント・docs には「なぜ」を書く (「何を」はコードが語る)。既存ファイルの
  コメント密度・言語 (日本語) に合わせる

## 5. 禁止事項

1. **D1 直接 INSERT** (上記)
2. **DSL への eval 文字列・チューリング完全化** (dsl.ts 冒頭コメントの設計意図)
3. **データ捏造**: 取得できない外部データ (例: FOMC 過去日程) を推測で埋めない。
   空のまま出荷し、理由をコード内コメントと HANDOFF に書く (events_backfill.py の前例)
4. **lookahead**: 同バー約定・未来参照 feature。fwd_ret_* を feature_defs に登録しない
   (docs/19 S-100 設計判断-4)
5. **verdict 判定への AI 利用** (docs/07 冒頭原則 — AIは仮説生成と要約のみ)
6. **チャット履歴を根拠にした実装**: リポジトリに書かれていない「決定」は存在しない。
   見つけたら HANDOFF の Open Questions に「文書化されていない前提」として起票
7. **凍結文書への追記**: docs/15 (凍結)。ロードマップは docs/18、タスクは docs/19、監査は docs/17
8. **docs のファイル名変更・移動** (§2)

## 6. 設計変更ルール

- **軽微** (誤記・実装との不一致の是正): 該当 docs を直接修正 + コミットメッセージに理由
- **実質的** (仕様・優先順位・アーキテクチャ): CTO のみが行う。手順 =
  該当 docs に日付付き追記 (元の記述は取り消し線か「旧」注記で残す — 履歴を本文でも追える) +
  docs/17 §6 に ADR 1行 + 影響するカード (docs/19) の更新
- Engineer が実装中に設計の誤りを発見した場合: 実装で勝手に直さず、
  (a) 明白なバグ (S-94 級) は修正 + docs 是正 + 実行ログに根拠を書く、
  (b) 設計判断の変更が要るものは HANDOFF に起票して CTO 判断を待つ
- ADR は覆してよい。ただし新 ADR で旧 ADR を明示的に参照して覆す (黙って矛盾させない)

## 7. レビュー方針

- Engineer の自己検証 = docs/19 §0-4 (機械検証 + 本番実測) が最低ライン
- CTO 受入 = カードの「受入条件」を本番またはテストで実測。コード品質は
  「既存パターンとの一致」を主眼にレビュー (美的リファクタは求めない)
- バグ修正カードには**必ず回帰テスト** (S-94/95/96 の前例: 「このバグが再発したら落ちるテスト」)
- 人間レビューが必要なもの: 本番UIの操作確認、secrets 投入、課金判断、外部データの真正性

## 8. GitHub 運用

- **ブランチ**: 開発は designated branch (`claude/cryptoedge-lab-design-6pzbel`)。
  main への反映は fast-forward のみ (マージコミットを作らない)。main 反映は
  オーナー承認済みの定常運用 (deploy.yml が main から本番デプロイ)
- **コミット**: 形式 `type(scope): S-XX -- 要約` (例: `feat(web,api): S-98 -- Action Queue ...`、
  docs のみは `docs: ...`)。1コミット = 1カード分を原則とし、本文に設計判断の要点を書く
- **push 後の義務**: ci.yml / deploy.yml の成功確認まで がセッションの仕事。
  失敗したら直すか、直せない理由を HANDOFF の Blockers に書く
- **本番検証**: デプロイ後、D1 読み取りまたは UI 操作で受入条件を実測 (docs/19 §0-4)
- **workflow_dispatch**: 手動ジョブ (events-backfill, lake-sync 等) の本番実行は
  オーナー承認を得てから。実行結果 (件数等) を実行ログに残す

## 9. セッションの開始と終了 (HANDOFF プロトコル)

- **開始時**: CLAUDE.md → docs/HANDOFF.md → 本書、の順で読む。HANDOFF の
  Current Card と自分の役割が一致することを確認してから作業する
- **終了時**: docs/HANDOFF.md を現在の真実に**上書き更新**してからコミット・push する。
  これを怠ると次セッションが壊れる — 本書で最も重要な規則
- HANDOFF に書ききれない詳細な経緯は docs/19 の実行ログへ。HANDOFF は常に
  「次のセッションが10分で現在地に立てる」密度を保つ (肥大化したら削る)
